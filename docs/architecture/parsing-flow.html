Spenso Parsing Flow

This document describes how symbolic atoms are parsed into tensor networks in crates/spenso/src/network/parsing. It focuses on the control flow, shorthand expansion, opaque leaves, structure inference, and edge cases that affect Schoonschip-style notation.

Entry Points

Parsing starts from NetworkParse methods such as parse_to_atom_net. The parser creates a fresh ParseState and calls try_from_view_impl on the input AtomView.

ParseState Tracks recursion depth and owns the dummy-index allocator used by shorthand materialization.
Tensor library Maps parsed structures to library tensors when a matching key exists.
Function library Used by the opaque tensor-expression boundary for target tensor types that need it.
ParseSettings Controls scalar precontraction, depth leaves, shorthand mode, and sum parsing behavior.

Settings

precontract_scalars When parsing products or sums, combine factors/summands that parse as pure scalars instead of keeping them as separate graph nodes.
take_first_term_from_sum Parse only the first summand. This is mainly for structure discovery when all summands are expected to expose the same slots.
depth_limit Turns composite expressions into leaves once the parser reaches the configured depth. The current checks live on add, mul, and pow parsing.
depth_is_product_depth When true, only products increment depth. When false, sums and powers increment depth too.
shorthand_parsing Expand lowers known shorthand into explicit graph structure. Opaque keeps supported shorthand as one leaf and infers its exposed structure.
parse_composite_scalars_as_tensors Passed into structure parsers that want to keep composite scalar expressions inspectable instead of immediately storing them as pure scalars.

Option Examples

ShorthandParsing::Expand F(p(mink(D)), mink(D,i)) is rewritten before generic leaf parsing as F(mink(D,d), mink(D,i)) * p(mink(D,d)), where d is a fresh dummy.
ShorthandParsing::Opaque { Fast } chain(bis(D,i), bis(D,j), F(in,out)) becomes one leaf. Its visible structure is inferred syntactically as the two endpoints bis(D,i) and bis(D,j).
ShorthandParsing::Opaque { Expanded } The same chain is expanded in a temporary network only to read dangling slots. The final parse still stores one opaque leaf.
precontract_scalars = true In (a + b) * F(mink(D,i)), a + b can be kept as one scalar factor next to the tensor network instead of a separate graph node.
take_first_term_from_sum = true F(mink(D,i)) + G(mink(D,i)) is represented by the first summand's structure. Use this only when the caller already knows the summands expose the same slots.
depth_limit = Some(1) A nested product can be handed to Parse::parse_with_settings as a leaf instead of recursively expanding every child. This is the same leaf boundary used by opaque parsing.
depth_is_product_depth = false Sums and powers count toward the depth limit too. With the default true, only product nesting increases parse depth.

Main Dispatch

try_from_view_impl dispatches by atom kind: 

Mul  -> try_from_mul
Fun  -> try_from_fun
Add  -> try_from_add
Pow  -> try_from_pow
else -> scalar fallback

Products, sums, and powers may hit the depth leaf boundary before they recurse. Function parsing is more symbol-specific: ordinary functions are parsed as leaves unless they are special syntax or contain expandable shorthand.

    AtomView::Pow(p) => Self::try_from_pow(...),
    a => Ok(Network::from_scalar(a.try_into()?)),
}

Chain And Trace

chain

parse_chain(chain(start, end, factors...))

  1. Parse start and end as concrete slots.
  2. Branch: no factors parses g(start, end).
  3. Branch: factors allocate links, replace in/out, materialize shorthand, parse, and multiply.
trace

parse_trace(trace(rep, factors...))

  1. Parse rep as a representation.
  2. Branch: no factors parses dim(rep).
  3. Branch: one factor closes with a bridge slot and two metrics.
  4. Branch: many factors allocate cyclic links, materialize, parse, and multiply.

Function Flow

Function parsing has the most important ordering rules.

  1. If shorthand_parsing is Opaque and the symbol is known shorthand, parse the whole function as one inferred leaf. Current known opaque shorthand symbols are chain, trace, and dot.
  2. Handle special expand-mode syntax: bracket, chain, trace, pure_scalar, and tagged wrappers.
  3. For generic functions in Expand mode, run shorthand materialization before ordinary leaf parsing.
  4. Try S::parse_with_settings. If it succeeds, use a library tensor when possible, otherwise concretize a tensor shell.
  5. If structure parsing fails, parse the function as a scalar expression.

Important: materialization must run before the generic leaf parser. Otherwise F(p(rep), slot) can be accepted as a lower-rank tensor with p(rep) stored as metadata, losing the required p(dummy) factor.

Expand vs Opaque

Expand

Shorthand is rewritten into explicit parser input. Chain and trace allocate internal links. Compact vector arguments allocate fresh slots. The parser then recurses on the rewritten expression.

Opaque

Supported shorthand roots become one leaf. Structure is inferred first, then TensorFromExpression constructs the target tensor from the original expression and inferred structure.

Opaque mode is checked before chain or trace expansion. Expand mode only materializes shorthand at explicit expansion points.

Opaque Leaf Boundary

Opaque shorthand parsing is split into two steps.

  1. StructureFromAtom::structure_from_atom determines the exposed slots. Fast uses a syntactic walk. Expanded builds an expanded shorthand network and reads dangling slots.
  2. TensorFromExpression::tensor_from_expression turns the original atom plus inferred structure into a tensor. The default implementation concretizes a shell from the structure. Specialized tensor types can preserve the expression or execute an expanded sub-network here.

Structure Inference

Fast structure inference is syntactic and intentionally non-semantic.

Function Direct slot arguments expose slots. aind(...) bundles are flattened. Other arguments are metadata.
Product Merge the structures of all factors. Scalar factors contribute an empty structure.
Sum Use the first summand's structure. The parser separately checks compatibility when it parses full sums.
Power Scalars stay scalar. Even powers of fully self-dual tensors become scalar. Odd integer powers keep the base structure. Other tensor powers are rejected.
chain Open endpoints are external slots. Factor slots are scanned recursively. in and out placeholders are wiring labels, not visible slots.
trace The representation argument is not an external slot. Factor slots are scanned recursively.

Materialization

ShorthandMaterializer lowers compact parser syntax into ordinary parser syntax. It stores a current atom plus additional_factors. Additional factors are multiplied beside the current atom and are not inspected by the materializer during that same pass; they are parsed when the parser recurses on the complete expression.

Compact Vector Argument

F(..., p(rep), ...)
-> F(..., rep(dummy), ...) * p(rep(dummy))

The dummy slot is allocated from the active ParseState, so it shares the namespace of the surrounding network parse.

Compact Scalar Product

g(p(rep), q(rep))
-> p(rep(dummy)) * q(rep(dummy))

This currently requires both compact vectors to have the same self-dual representation. dot(p(rep), q(rep)) follows the same rule.

Compact Vector Detection

A compact vector is a function that:

  • is not metric or dot;
  • is not itself a representation;
  • has no explicit slot argument;
  • has exactly one direct argument matching the representation wildcard convention.

Sums are accepted only when every summand is a compact vector with the same representation. Products and powers are not compact vector syntax.

Chain Expansion

In expand mode, chain(start, end, factors...) expects concrete endpoint slots.

  1. With no factors, it becomes g(start, end).
  2. Each factor gets a left and right link slot.
  3. in and out placeholders are replaced syntactically.
  4. The factor is passed through shorthand materialization.
  5. Each materialized factor is parsed into one or more factor networks.
  6. The factor networks are multiplied in chain order.

If a materialized chain factor is a product with scalar factors and exactly one tensor factor, scalar factors are split out so they can be handled separately. Otherwise the whole factor product is parsed as one expression.

crates/spenso/src/network/parsing/mod.rs:647 

let start = Slot::try_from(args[0])?;
let end = Slot::try_from(args[1])?;
...
let factor = ChainExpansion::replace_placeholders(...);
let factor = ShorthandMaterializer::new(&state, settings)
    .materialize_shorthand(factor.as_view())
    .unwrap_or(factor);

Trace Expansion

In expand mode, trace(rep, factors...) starts from a representation, not endpoint slots.

  1. No factors: parse the representation dimension.
  2. One factor: allocate left, right, and bridge slots, then close the factor with two metrics.
  3. Multiple factors: allocate cyclic links and replace each factor's placeholders with adjacent links.
  4. Materialize shorthand inside each factor before parsing it.
  5. Multiply the factor networks.

let rep = Representation::try_from(*rep_view)?;

Products, Sums, Powers

Product Parse every factor recursively. If scalar precontraction is enabled, pure scalar factors are combined. The old special case p(rep) * q(rep) is intentionally not a shorthand materialization path.
Sum Parse summands recursively and require compatible network states. Pure scalar summands may be precontracted into one scalar sum.
Power Integer powers recurse into the base. Tensor powers are allowed only when the network state supports the exponent. Non-integer powers fall back to scalar parsing.

Edge Cases

F(p(rep), slot) Must materialize before ordinary function parsing. Expected expansion is F(dummy, slot) * p(dummy), leaving slot external and the dummy internal.
p(rep) A standalone compact vector root is not materialized. It can only become a vector when used in a materializable argument or scalar-product position.
p(rep) * q(rep) Not valid shorthand in the parser. The product path does not materialize this into a dot product.
g(slot, p(rep)) Materializes to g(slot, dummy) * p(dummy). The contraction is represented by the network, not by a pre-parser algebra rewrite.
g(p(rep), q(rep)) For matching self-dual reps, materializes to p(dummy) * q(dummy).
Dualizable compact scalar products Current gap: compact scalar-product materialization requires self-duality. Dualizable cases produced by algebraic Schoonschip rules are not handled by this specific materializer rule.
Compact vector endpoints in chains Expand-mode chain endpoints are parsed before factor materialization and must be concrete slots. A compact vector endpoint from an algebraic pre-pass is not accepted as a chain endpoint by this parser path.
Metadata false positives The compact-vector convention is syntactic. Any function argument with exactly one direct representation argument and no slots can be treated as a compact vector in expand mode.
Opaque fast vs expanded inference Fast inference is cheap and syntactic. Expanded inference builds an expanded network and reads dangling slots, so it is a validation oracle but more expensive.
Depth leaves Depth leaves use Parse::parse_with_settings first and scalar fallback second. They do not run chain or trace expansion at that boundary.

Short Algorithm

parse(atom):
  dispatch by atom kind

parse_fun(fun):
  if opaque mode and fun is chain/trace/dot:
    infer structure
    build opaque tensor leaf from original expression

  if fun is chain or trace in expand mode:
    allocate links
    replace in/out placeholders
    materialize shorthand inside factors
    parse factor expressions

  if fun is ordinary and expand mode:
    if materialize_shorthand(fun) changed it:
      parse(materialized expression)

  if ordinary structure parsing succeeds:
    return library tensor or concretized shell

  return scalar expression

materialize_shorthand(fun):
  if fun is compact metric/dot:
    return current = 1, additional_factors = [lhs(dummy), rhs(dummy)]

  scan arguments:
    compact vector argument -> current slot plus additional vector factor
    nested function argument -> recurse
    other argument -> unchanged

  return rebuilt current multiplied by additional_factors
